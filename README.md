# 駅徒歩 体感指標

不動産広告の「駅徒歩◯分（80m/分）」を、勾配・階段・信号・荷重で補正して物件を比較するための単一ファイル Web アプリ。六甲〜阪神間のような坂地形で、広告の徒歩分数が実態とどれだけ乖離するかを数値化する。

ビルド不要。`index.html` ひとつで動く（外部依存なし）。

## 使い方

1. `index.html` をブラウザで開く
2. 物件名・道路距離・高低差を入力（起伏のある道は「片道の登り累計／下り累計標高」を入れると精度が上がる）
3. 現地補正（階段・信号・踏切・ベビーカー・抱っこ・主観補正）を入れる
4. 「比較表に追加」で物件を蓄積。体感分数の小さい順に自動で並ぶ

入力した物件は同じブラウザの localStorage に保存される。さらに集計を有効化すると（下記「データを集計する」）、追加した算出データを全員分まとめて Google スプレッドシートに蓄積できる。

## 指標の定義（往復・起伏ベース）

起伏のある道は、片道の **登り累計標高 `Au`** と **下り累計標高 `Ad`** を入力する。往復では上下が入れ替わるため、**往復の総登り＝総下り＝`V = Au + Ad`**（net高低差 `Au − Ad` は相殺されて往復エネルギーには効かない）。登り/下り累計を入れない場合は高低差(net)を登りのみ（`Au=高低差, Ad=0`）として扱う。

ACSM歩行式の登り項を道のりで積分すると **鉛直コストは累計標高だけで決まる**（距離・速度に非依存）。これを使って往復を評価する。

| 指標 | 式 |
|---|---|
| 法定徒歩分数（片道） | `道路距離 ÷ 80m`（端数切上げ）。広告の徒歩分の基準値 |
| 運動強度 METs（往復平均） | 往復の総VO₂ ÷ 往復時間 ÷ 3.5。総VO₂/kg = `(0.1·S+3.5)·往復分 + 1.8·V·(1+0.06)`（S=速度 m/分） |
| 体感分数（往復） | `(法定往復 + V/10) × 荷重倍率 ＋ 待ち時間×2`（V/10 は Naismith：+1分/10m昇） |
| 消費カロリー（往復） | `(平地METs×体重×往復時間h + 鉛直VO₂×体重×0.005) × 荷重倍率`（5 kcal/L O₂） |
| コース定数（往復） | `1.8·往復時間(h) + 0.3·2D(km) + 10·V(km) + 0.6·V(km)` |

- **荷重倍率** = 現地補正 × ベビーカー × 抱っこ（各係数は画面の「係数の設定」で調整可）
- **階段** = `段数 × 1段の高さ(既定0.17m)` を片道の登り累計 `Au` に加算
- **下りコスト** = 登りの 6%（`DOWN_F=0.06`、コース定数 0.6/10 の比に整合）。緩い下りの労力を軽く加味
- **待ち時間** = （信号回数 × 想定待ち ＋ 踏切回数 × 想定待ち）× 2（往復）
- **体重は消費カロリーにしか効かない。** METs は体重で正規化された量なので、体感分数・判定バンドは体重と無関係。

判定バンド（往復平均METs × 荷重倍率）: 〜3.5 楽 / 〜5 標準 / 〜6.5 ややきつい / それ以上 きつい

## 前提と限界

- 下りは登りの6%の軽いコストとして簡略化（セグメント勾配ごとの厳密な代謝モデルではない）
- 実際の登坂は表示時間より遅くなる（速度は固定）
- コース定数は登山向けの式で、数百mの短距離では登り項（×10）を過大評価しがち。絶対値より物件間の相対順位として使う
- 道路距離は地図実測、累計標高は地理院地図の断面ツールなどから取得すると精度が上がる

## GitHub Pages で公開する

```bash
git init
git add index.html README.md
git commit -m "init: 駅徒歩 体感指標"
git branch -M main
git remote add origin git@github.com:<user>/<repo>.git
git push -u origin main
```

リポジトリの Settings → Pages → Source を `main` / `/ (root)` に設定すると `https://<user>.github.io/<repo>/` で公開される。

## データを集計する（Googleスプレッドシート）

公開ページで「比較表に追加」されるたびに、算出データ（＋入力値・日時）を全員分まとめて
Google スプレッドシートに蓄積できる。サーバー不要・無料。`apps-script/Code.gs` を使う。

1. 集計用の Google スプレッドシートを新規作成
2. **拡張機能 → Apps Script** を開き、`apps-script/Code.gs` の内容を貼り付けて保存
3. **デプロイ → 新しいデプロイ → 種類「ウェブアプリ」**
   - 実行ユーザー: **自分**
   - アクセスできるユーザー: **全員**
4. 発行された `https://script.google.com/macros/s/.../exec` URL を、`index.html` の
   `LOG_ENDPOINT`（`<script>` 冒頭の定数）に貼る
5. commit して push → GitHub Pages に反映

これで公開ページから追加された行が、別端末・別ユーザー分も含めて同じスプレッドシートの
`log` シートに1行ずつ溜まる。シートが空のときは1行目に列見出しが自動で入る。

送信は失敗してもページ側の動作には影響しない（オフラインや URL 誤りでも比較表追加は成功する）。
`LOG_ENDPOINT` を空のままにすれば送信は行われず、従来どおり localStorage のみで動く。

### プライバシー

- 集計を有効化すると、追加時に算出データが匿名でスプレッドシートに記録される（個人を識別する
  情報は送らない）。物件名は任意項目なので、**個人名・住所など個人情報は入力しない**こと。
- 有効化中は入力カードにその旨の注記が表示される。

## 徒歩ルート地図を表示する（Google Maps Embed API）

「徒歩ルート（Googleマップ）」カードに徒歩ルートを表示できる。カード内の **出発地・目的地の入力欄**
に住所などを直接入力して経路を指定でき、未入力なら物件名→最寄駅で自動表示する。
**Maps Embed API は無料**。`index.html` の `MAPS_EMBED_KEY` が空のうちは設定手順の注記のみ表示。

1. [Google Cloud Console](https://console.cloud.google.com/) でプロジェクトを作成
2. **APIとサービス → ライブラリ → 「Maps Embed API」を有効化**
3. **認証情報 → 認証情報を作成 → APIキー**
4. 作成したキーに **アプリケーションの制限 → HTTPリファラー** を設定し、
   `https://nobushii.github.io/*` を許可（キー悪用を防ぐ）。
   API制限で「Maps Embed API」のみに絞るとより安全。
5. そのキーを `index.html` の `MAPS_EMBED_KEY`（`<script>` 冒頭の定数）に貼る → push

表示された徒歩距離を見て「駅までの道路距離」に手入力する用途（距離の自動入力はしない）。
キーは Embed API 限定＋リファラ制限のため、公開ページに置いても実害は小さい。
